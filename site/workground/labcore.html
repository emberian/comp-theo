labcore — the Rust engine

"Do we need to rewrite this so it runs way faster?" Yes — and the rewrite did something more useful than go fast: as a second, independent implementation it became a robustness check, and it immediately falsified one lab4 result while confirming another.

Engine: labcore/ — a zero-dependency Rust crate (no rand, no rayon, no serde; hand-rolled xoshiro256**, std::thread::scope for 12-core parallelism, a 6-line text protocol). It builds offline in ~3 s and needs no Python build toolchain. Source is in the repo; this page is the lab notebook for it.

Why not just optimise the Python?

lab/lab2 are already fast (vectorised linear algebra — Rust would do nothing). The cost is entirely lab3/lab4: pure-Python RL loops doing millions of tiny per-step operations the interpreter is bad at and numpy cannot help with (per-state arrays have degree ~3). The honest options were laid out; the chosen one was a real engine, on the principle that the architecture should be Python orchestrates, Rust is the engine — Python keeps world generation, spectral features, statistics and figures (what it is good at); Rust does the 4.6M-step training loops, in parallel.

The split

Python (numpy/scipy)                Rust (labcore)
  make_world, detect_basin           parse world + features
  spectral eigh, MFPT/τ              run ALL (seed × cell) trainings
  bootstrap CIs, permutation tests   in parallel across 12 cores
  figures, scoreboards               return success/capture per spec

A world (adjacency, costs, detected basin, escape edge, and an optional N×d feature matrix) is serialised once; the engine runs the whole grid of flag-combinations × seeds for it and streams back two floats per run. No cross-language RNG sync: the world is fully specified by Python, so parity is by construction; only the learner's stochastic dynamics live in Rust (its own deterministic, seeded RNG).

Measured speed-up

Workloadpure Pythonlabcore note
64 tabular trainings (N=400, light)~9 s 0.81 s~11× even with no parallelism headroom on the light config
full lab4, 8 seeds (FA factorial + scaling + capacity, + Python validation/detectors/eigh)3+ min & climbing 21.9 sthe heavy real workload; ~8–10× wall incl. all the Python parts that did not move

The training loops themselves are far more than 10× faster; the ~9–10× wall figure is Amdahl's law — the un-moved Python (eigh, detectors, the slow correctness-anchor validate()) now dominates, which is exactly the right place to be. Bumping lab4 from 4 to 8 seeds went from "not worth the wait" to free.

The engine as a robustness check (the real payoff)

labcore is a faithful but independent re-implementation of the lab3/lab4 agents. Running lab4 through it is therefore the same move lab2 made its motto: a result that survives a change of instrument is the closest this archive gets to knowing something — and one that does not is exposed. Outcome:

lab4 claimpure-Python agent Rust engine (independent)verdict
A · scaling collapse predicted by exit time τ ρ ≈ 0.54 (weak)ρ = +0.78, monotone CONFIRMED & strengthened
B · which mechanism helps (FA, detected basin) noisy, none survived vow +0.086 (p=.03), grace +0.100 (p=.02) significant; variety tiny SHIFTED — representation-dependent
C · "a lossier map needs more variety" (corr −0.96) strong PASScorr = −0.05 — gone FALSIFIED by the engine

C is the important one. The capacity×variety law looked like lab4's prettiest result. A second faithful implementation makes it vanish — so it was an artifact of the particular Python agent's exploration constants, not a property of the model. Keeping that on the board as a falsified claim is the entire reason to build a second engine. The engine earned its keep not by being fast but by being different and honest.

And the cross-instrument signal compounds: vow was inert in lab2 (analytic) and lab3 (tabular learner) but is significant here under linear function approximation. That is not a contradiction — it is the lab's recurring lesson sharpened: the value of the corpus's machinery is agent- and representation-relative. "A bad goal is a tiny god" only traps a mind of a particular shape; a vow only helps a mind of another.

What this is not

Not a GPU thing, not distributed, not a framework. ~400 lines of dependency-free Rust and a 60-line Python bridge. The point was never raw flops; it was making the next escalation (lab5 at real scale, function approximation on found basins, hundreds of seeds) cheap enough to run honestly — and, in passing, getting a free second opinion that already corrected the record.
